mtcars[row_index, ]

grade_result(
  pass_if(~ identical(.result, mtcars[1:2, ]))

Enter Tidy R

Up to this point we have been using functions and operations available in “base” R—what you get from R out of the box. These include operators like [ or [[ and functions like head. Although these operators and functions are fine, they can be clunky to use and don’t handle missing values and out-of-bound vectors properly selecting subsets of the dataframe.

The tidyverse is a collection of packages that is designed to handle operations in a safe and consistent manner, while providing intuitive functions and data workflow using the %>% ‘pipe’ operator from the maggitr package. If the base R looks like this:

result1 = function1(data, ...)
result2 = function2(result1, ...)
result3 = function3(result2, ...)

or in the nested form like this:

result = function3(
             function2(
                 function1(data, ...))
             )
         )

The tidy R offers a fluent interface style which looks like this:

results <- data %>%
             function1(...) %>%
             function2(...) %>%
             function3(...)

Note how the %>% allows one to pipe the data and/or result of a function to the next function, which you can think of as method chaining in Python. This makes the function evaluation order clearer as it goes from top to bottom or left to right, which is natural compared to the nested call style. The style also is less tedious than assigning to temporary variables.

Also keep in mind one could choose not to store the result, in case you just want to apply the functions:

data %>%
  function1(...) %>%
  function2(...) %>%
  function3(...)

These functions perform actions on the data so we call them ‘verbs’. There are a few you should become familiar with especially from the dplyr package, which defines this style of programming:

  • slice: choose observations (rows) by row numbers
  • select: choose variables (columns) by name
  • filter: choose observations (rows) by value(s)
  • arrange: reorder rows
  • mutate: derive new variables from existing ones
  • group_by: define subsets of rows for further processing
  • summarize: combine many values to create a single new value

There are also variations for some of these but we will only cover these main verbs. All of these verbs follow the form:

verb(data, args)

The first argument is always the data being piped forward from the previous function. However, it is important to note that passing the data is not necessary when already piped forward from previous data or operation:

data %>% verb1(args) %>% verb2(args)

So what?

At this point you might be thinking, “Isn’t this just a different style? How does it really benefit me?” It’s true that style can be subjective but we’ll go over some nice conveniences that come with this style.

If you want to learn more:

The idea of “tidy” data was introduced by Hadley Wickham in his paper in the Journal of Statistical Software and is explained in detail in the R for Data Science book as well. It’s worth the read if you are interested in the deeper roots of this way of programming.

slice

The problem with slicing using [

Let’s see what happens when we subset rows of df with a range that clearly exceeds the total number of rows.

In Python, there are no surprises.

df[1:10]
     a    b    c
1  2.0  4.0  7.0
2  NaN  5.0  8.0

However, execute the same operation in R:

df[1:10, ]

GOTCHA! You should see some extra rows full of missing values or NAs.

This results from a concept called vector recycling. In a nutshell, R prefers performing operations on equal sized vectors and will “recycle” the shorter one to meet the length of the bigger one—fair enough! The shorter vector in our case is the number of rows df has which we know to be 3:

nrow(df)
[1] 3

But the range we specified to subset is 1:10, a vector that is 10 elements long. So R recycles the 3 until reaching the 10, but it will create rows with NA which makes sense because it has no idea what you want the values to be! But, this is certainly confusing coming from Python.

Tinker with your code above and change the range so that it exceeds 3 and you’ll see it pads as many rows as required.

Here’s where our first tidy verb slice comes in handy, which behaves as you’d expect in Python. Try out slice(df, start:end) where df is your dataframe, start:end is a range for the rows you want from df. It should make you happy with the results: 
slice(df, start:end)

Sadly, our story does not end here with [. Let’s explore other problems in the next topic.

select

Another handy verb is select which allows us to select columns much more easily than using [ and will even allow us to deselect columns as well. Let’s select the am (transmission) and gear columns of mtcars:

The syntax is select(data, cols) where the data is a dataframe or tibble and the columns can be either comma-separated labels, a list of labels using c() or indices. Let’s look at the column names first:

select(mtcars, am, gear)

Note how the column labels after the mtcars do not have to be quoted. You can also make use of that %>% syntax so we only print the first few rows if you want. Examine the code below and note that labels can be quoted as well:

select(mtcars, 'am', 'gear') %>% head()

In fact, select is flexible enough to allow using c() where the items are not quoted:

select(mtcars, c(am, gear)) %>% head()
but here also, the items can be quoted: 
select(mtcars, c('am', 'gear')) %>% head()

So what?!

Isn’t all this just a different syntax that pretty much does the same thing as [? Well, yes but including the flexibility shown above, select has some other nice capabilities that we don’t get from [.

filter

The problem with filtering in base R

Now let’s look at filtering with the a simple dataframe called df:

The problem with filtering using [

In Python, we can safely use [ to filter df for rows where a is greater than 1

df[df.a > 1]
     a    b    c
1  2.0  4.0  7.0

and also where b is greater than 1:

df[df.b > 1]
     a    b    c
1  2.0  4.0  7.0
2  NaN  5.0  8.0

Now in R, try using [ to filter df for rows where a is greater than 1 notice what happens:

Now, try using [ to filter df for rows where b is greater than 1:

GOTCHA! If you don’t see it yet, the issue for both of these operations is that they both kept rows with NA even though we wouldn’t normally see them in Python.

Enter filter

Our next verb filter addresses the issues with filtering using [.

The syntax is: filter(df, condition)

filter(df, a > 1)
filter(df, b > 1)

Note how the column in the condition does not need to be referenced from the data using $

Notice also how we don’t run into any issues now even with NA in our dataframe, thereby mimicing Python’s behavior.

Try playing filter yourself:

Depending on what you tried, you may have run into some confusion about all of its capabilities like multiple conditions etc. which we will now cover.

Multiple conditions

What if we want to apply multiple conditions? Let’s say we’re looking to buy one of those cars in mtacrs dataset and we want cars such that their mpg is greater than 20 and there are 6 cyl or cylinders.

Think back to using query in Python:

mtcars.query('mpg > 20 & cyl == 6')
                 mpg  cyl   disp     hp  drat  ...   qsec   vs   am  gear  carb
Mazda RX4       21.0  6.0  160.0  110.0  3.90  ...  16.46  0.0  1.0   4.0   4.0
Mazda RX4 Wag   21.0  6.0  160.0  110.0  3.90  ...  17.02  0.0  1.0   4.0   4.0
Hornet 4 Drive  21.4  6.0  258.0  110.0  3.08  ...  19.44  1.0  0.0   3.0   1.0

[3 rows x 11 columns]

We can mimic Python’s query using R’s filter to do the same operation, which also looks syntactically similar:

filter(mtcars, mpg > 20 & cyl == 6)

The difference is that the conditions don’t need to be quoted. filter also allows using , to indicate the & boolean operator. Note that this is the only shorthand and just applies to & not for other operators like |. Execute the following:

filter(mtcars, mpg > 20, cyl == 6)

GOTCHA! You might have noticed something different for both outputs compared to Python. Where are the rownames?!

The one small downside of filter and some other verbs is that rownames can be silently dropped. Often times, this doesn’t matter for analysis.

However, we have a way to preserve the names using the rownames_to_column function from the tibble package within the tidyverse package. If you imported tidyverse there is no need to import it.

Fix the code below by calling rownames_to_column to our data in the first argument:

filter(mtcars, mpg > 20, cyl == 6)
grade_result(
  pass_if(~ identical(.result, filter(rownames_to_column(mtcars), mpg > 20, cyl == 6)), "We got our rownames back!"),
  fail_if(~ identical(.result, filter(mtcars, mpg > 20, cyl == 6)), "Oops! Still missing the rownames.")
)

There you have it! We have covered the basics of filter, an important verb in our data analysis toolkit!

There is a ton you can do with filter and we will come back to it in our last topic, but you can always dig into the docs to learn more!

In R, there are a couple of ways to sort. First let’s look at the base R way.

You can use the order function:

mtcars[order(mtcars$mpg),]

and unlike Python, for descending order, you merely prepend the column reference with -:

mtcars[order(-mtcars$mpg),]

However, it’s a little clunky to have to reference the column using $ everytime. What if we left out the mtcars$ part?

Try executing the code below:

mtcars[order(mpg), ]

GOTCHA! R is confused what the mpg is when directly referencing it without the use of $! You should’ve seen an error:

Error: Must use a vector in `[`, not an object of class matrix.

There is a way to avoid this by first using attach which you can think of as adding a path for R to search for. attach allows you to add mtcars into the search path, which means it helps R figure out that you meant the mpg column without direct reference:

attach(mtcars)
mtcars[order(mpg), ]

and descending:

attach(mtcars)
mtcars[order(-mpg), ]

Note how we didn’t have to reference the mpg using the $ anymore. But that’s one extra function just to achieve direct reference, just so we can sort easily. There must be a better way, something like Python’s sort_values!

Enter arrange:

Using arrange we can directly mention the columns we want to sort the rows by:

arrange(mtcars, mpg)

then for descending we prepend a - like we did for order:

arrange(mtcars, -mpg)

If the - is confusing, there is a convenient function called desc() which can be used to sort by descending order:

arrange(mtcars, desc(mpg))

The - and desc do the same thing but the latter is clearer in intention.

GOTCHA! If you haven’t noticed yet, unlike the order function used in base R, the rownames are silently dropped when using arrange. As mentioned before, if row names are important to you, use the rownames_to_column() function.

Let’s do that now. Go ahead and use rownames_to_column() on mtcars, then make use of that %>% to pipe the result to arrange, sorting by mpg descending order. Finally, pipe the result to a head as we only want to gander at the first few rows:

rownames_to_column(...) %>% 
  verb(...) %>% 
  verb(...)
grade_result(
  pass_if(~ identical(., rownames_to_column(mtcars) %>% arrange(-mpg) %>% head()), "")
)

Like Python, there are two ways we can drop a column in R. One is using the select verb and the other is setting [. One difference using the [ for R is one simply needs to set the column to null:

First let’s look at the base R way. Like Python, you can use the [ to delete a column:

mtcars$mpg <- NULL
mtcars$disp <- mtcars$disp * 0.0163871
head(mtcars)

But unlike Python’s assign, we can use our mutate to both drop and modify columns:

mtcars %>% mutate(mpg = NULL, disp = disp * 0.0163871) %>% head()

group_by

So far we have been using the tidy verbs mostly in isolation for the purpose of learn how each one works. But, it’s now time to explore performing operations on groups defined by variables. Using a handy verb called group_by, we’ll cover how it changes interactions with other verbs as well as how it relates to Python.

by_cyl <- mtcars %>% group_by(cyl)
by_cyl

Select

by_cyl %>% select(group_cols())

Filter

# for each cyl, give me row of the max disp
by_cyl %>% filter(disp == max(disp))

Arrange

GOTCHA! Grouped arrange ignores groups:

by_cyl %>% arrange(desc(disp))

unless you specifically ask:

by_cyl %>% arrange(desc(wt), .by_group = TRUE)

Mutate

by_cyl %>% mutate(rank = min_rank(desc(mpg)))

Let’s make it easier to see the effects by moving rank up to the front and arrangeing by rank.

# for each cyl, give me row of the max disp
by_cyl %>% 
  mutate(rank = min_rank(desc(mpg))) %>% 
  select(rank, everything()) %>% 
  arrange(rank, .by_group = TRUE)

Now we can see the rows ordered according to the rank which is in turn based on mpg for each cyl groups 4, 6, and 8.

GOTCHA! Note that you can’t overwrite grouping variables:

gdf <- mtcars %>% group_by(cyl)
try(mutate(gdf, cyl = cyl * 100))

Executing this snippet you will see the error:

Error : Column `cyl` can't be modified because it's a grouping variable
